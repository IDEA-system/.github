## 1. 🏢 株式会社IDEA システム課 GitHub 運用ルール

ここは株式会社IDEAのシステム・インフラ管理、アプリケーション開発を行うための公式GitHub Organizationです。

---

## 2. 基本方針（権限の考え方）

| 観点 | 内容 |
|------|------|
| 閲覧の原則 | **基本的な閲覧権限はチーム単位で付与する**（個人にバラバラと付けないのが原則）。 |
| デフォルト | **個人ユーザーは、原則としてリポジトリを見られない**（必要なチームに入れる）。 |

### 2.1 権限の二層（標準 / 機密）【要記入】

| 区分 | `idea-writer` の付与 | 
|------|----------------------|
| 標準リポジトリ | 閲覧可能 |

---

## 3. 個人ユーザーと権限

| 項目 | ルール |
|------|--------|
| リポジトリ閲覧 | **基本は全て見えない**（アクセスが必要なら適切なチームに所属させる）。 |
| アカウントに付与するロール | **原則 `write`**（用途に応じて read 等に変更する場合はルールに沿う）。 |

### 3.1 「ロール」が指す範囲【要記入】

| 用語 | 本ルールでの意味（GitHub 上のどの設定か） |
|------|------------------------------------------|
| リポジトリ権限（read / write / admin 等） | write |

---

## 4. チーム

### 4.1 チームの位置づけ

- **社内用**と**社外（ベンダー）用**でチームを分ける。
- **社内ユーザーは `idea-writer` に所属する**（チーム作成ルールとセット）。

### 4.2 チーム作成ルール

| 区分 | ルール |
|------|--------|
| 社内(idea-writer) | IDEA 社員。　業務委託も含む |
| 社外 | **ベンダーごと**にチームを作成する。なお複数ベンダーがいる場合は会社ごとにチームを作成 |

### 4.3 命名規則（共通）

- **形式**: `組織名-ロール`（小文字・ハイフン区切りケバブケースを採用するものとする）。
- **例（社外）**: `newrelic-writer` のように **ベンダー名（小文字）-（権限）**。

### 4.4 社内チーム：`idea-writer`

| 項目 | 内容 |
|------|------|
| チーム名 | `idea-writer` |
| 閲覧・操作 | **全リポジトリにアクセス可能**。**リポジトリの作成も可能**。 |

### 4.5 社外（ベンダー）チーム

| 項目 | 内容 |
|------|------|
| ロール | 用途に合わせて **`read` または `write`** を付与する。 |
| 閲覧範囲 | **その PJ に関わるリポジトリのみ**（不要な広い権限は付けない）。 |

---

## 5. リポジトリ

### 5.1 誰がリポジトリを作れるか

- **`idea-writer` チームに所属しているユーザーのみ**、リポジトリを作成できる。

### 5.2 いつリポジトリを分けるか（作成・分割の判断）

| 観点 | 分ける・作る理由の例 |
|------|----------------------|
| **システム** | **アーキテクチャ上**、境界として分けた方がよい場合。 |
| **セキュリティ** | **特定の人にしか見せたくない**情報・コードがある場合。 |
| **運用・開発体験** | **誰が管理するか**を明確にしたい場合。 |

### 5.3 リポジトリ命名規則

- **ケバブケース**（例: `my-service-api`）。

### 5.4 権限まわりの再掲（リポジトリ単位）

- **個人ユーザー権限**: 基本、**リポジトリ単体では見えない**（チーム経由で付与）。
- **チーム**: 社内は **`idea-writer`**、ベンダーは **ベンダーごとのチーム**に振り分ける。

### 5.5 Organization メンバーのベース権限

| 項目 | 設定値・方針 |
|------|--------------|
| Base permission（組織のデフォルト閲覧権限） | write |
| メンバー招待・チーム追加の標準フロー | 平山、村木にスラックにて依頼 |

## 🔒 7. セキュリティ・機密情報の取り扱い（重要）

**パスワードやAPIキーなどの機密情報を、絶対にGitHub上にPushしないでください。**

- **`.env` ファイルでの管理**
  システム接続用のID、パスワード、アクセスキー等の管理情報は、ソースコードに直接書き込まず、必ず環境変数（`.env` ファイル等）として分離して管理してください。
- **`.gitignore` の徹底**
  `.env` ファイルや、ローカルの開発環境固有のパスワードが記載されたファイルは、必ずリポジトリ作成時に `.gitignore` に登録し、GitHubへのアップロード対象から除外してください。
